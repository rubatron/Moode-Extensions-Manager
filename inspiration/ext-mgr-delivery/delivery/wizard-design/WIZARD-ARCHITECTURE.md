## Sandbox philosophy

The extension folder is the single source of truth. Everything visible outside the
sandbox is a reference *back* into it — never the other way around.

```
/var/www/extensions/installed/<ext-id>/      ← SANDBOX
├── packages/services/   ──symlink──→   /etc/systemd/system/
├── packages/config/udev/──symlink──→   /etc/udev/rules.d/
├── packages/config/lirc/──symlink──→   /etc/lirc/
├── packages/bin/        ──symlink──→   /usr/local/bin/
├── packages/pylib/      (PYTHONPATH set in service unit)
│
└── data/install-footprint.json  ← every symlink/package/boot-line tracked here
```

Three reference mechanisms:

| Colour | Mechanism | Example |
|--------|-----------|---------|
| Green  | Symlink | `.service` units → `/etc/systemd/system/` |
| Grey   | Symlink (warning) | udev rules, binaries, lirc config |
| Orange | Append-only | `/boot/config.txt` with BEGIN/END guards |
| Red    | Blocked | `/var/www/` outside sandbox → violation |

`data/install-footprint.json` is what makes clean uninstall possible: it records
every symlink, apt package, pip package, and `/boot/` line added outside the sandbox.
`uninstall.sh` reads only the footprint — no guessing.

## Path policy (22 entries, full table in `ext_helper.py PATH_POLICY`)

| Severity | Prefix | Bundle target |

## Dev kit — building an importable zip

The dev kit ships `scripts/build-zip.sh` which produces a clean, import-ready zip.

```bash
bash scripts/build-zip.sh              # → <ext-id>-<version>.zip
bash scripts/build-zip.sh --dry-run    # list files without building
bash scripts/build-zip.sh -o /path     # custom output dir
```

VSCode: `Ctrl+Shift+B` → **ext-mgr: Build zip**

### What build-zip.sh does

1. Reads `id` and `version` from `manifest.json` → names zip `<ext-id>-<version>.zip`
2. Runs `ext_helper.py scan` — **aborts on violations**, prints warnings non-blocking
3. Builds file list via Python exclusion logic (no raw glob patterns)
4. Packages zip with `zipfile.ZipFile` — folder name preserved in archive

### Dev-only files — always excluded

| File/pattern | Reason |
|-------------|--------|
| `.vscode/` | VSCode workspace config — dev tooling |
| `scripts/build-zip.sh` | Dev tooling — excludes itself |
| `*.code-workspace` | VSCode workspace file |
| `.git/`, `.gitignore`, `.gitkeep` | VCS artefacts |
| `__pycache__/`, `*.pyc`, `*.pyo` | Python bytecode cache |
| `logs/` | Generated at install time |
| `cache/` | Generated at install time |
| `data/install-footprint.json` | Generated by install.sh |
| `.DS_Store`, `Thumbs.db` | OS junk |

### Import endpoint — second safety net

Even if a zip is built manually (not via `build-zip.sh`), the import endpoint
strips dev files after extraction:

```php
// wizard.php — strip_dev_files()
$DEV_PATTERNS = [
    '.vscode/', '.vscode', '*.code-workspace',
    '.git/', '.gitignore', '__pycache__/', '*.pyc',
    '.DS_Store', 'Thumbs.db',
    'scripts/build-zip.sh',
];
```

Defence in depth: `build-zip.sh` excludes → import endpoint strips as fallback.

## Wizard flow

```
Browser                  wizard.php              ext_helper.py
   │                         │                        │
   ├─ POST zip ─────────────→│                        │
   │                         ├─ extract to /tmp/ ─→  │
   │                         ├─ strip_dev_files()     │
   │                         ├─ scan ──────────────→ scan()
   │                         │ ←── JSON result ───────┤
   │ ←── prefilled wizard ───┤                        │
   │                         │                        │
   ├─ steps 1–6 ────────────→│                        │
   │                         │                        │
   ├─ POST generate ─────────│                        │
   │                         ├─ rewrite ───────────→ rewrite()
   │                         ├─ write manifest.json   │

   │ ←── application/zip ────┤                        │
```

## wizard.php endpoints

### POST ?action=upload
Input: `multipart/form-data` — `zip` field
Output:
```json
{ "ok": true, "session_id": "uuid", "scan": { ...ext_helper scan output... } }
```

### POST ?action=generate
Input: `application/json`
```json
{
  "session_id": "uuid", "ext_id": "my-extension",
  "name": "My Extension", "version": "1.0.0", "type": "functionality",
  "icon": "fa-solid fa-music", "menu_m": true, "menu_lib": true, "menu_sys": false,
  "settings_only": false, "stage": "visible-by-default",
  "svc_name": "my-extension.service", "deps": ["moode-extmgr.service"],
  "apt_packages": []
}
```
Output: `application/zip` — clean, ready to import

### POST ?action=upload-icon
Input: `multipart/form-data` — `icon` (.ico/.png/.svg, max 512KB) + `session_id`
Output: `{"ok": true, "path": "assets/images/icon.ico"}`

## stageProfile

| Value | Behaviour |
|-------|-----------|
| `visible-by-default` | Extension appears in configured menus immediately after install |
| `hidden-by-default` | Installed but hidden; user must manually activate |

Use `hidden-by-default` for extensions requiring pre-use configuration
(e.g. streaming services needing credentials, hardware needing GPIO setup).

## Security checklist

- [ ] Zip-slip: all entries must resolve inside target dir
- [ ] Session ID is UUID stored in `$_SESSION`
- [ ] Temp dirs cleaned after download or 30min TTL
- [ ] Zip max 50MB, icon max 512KB
- [ ] Icon extensions: `.ico`, `.png`, `.svg` only
- [ ] `manifest.id` sanitized to `[a-z0-9\-]`
- [ ] Violations abort generate (scanner must return clean)
- [ ] Dev files stripped via `strip_dev_files()` after extraction

## Import pipeline (after wizard generates zip)

```
User uploads zip to ext-mgr UI
          ↓
Zip-slip validation
          ↓
Extract → /var/www/extensions/staging/<ext-id>/
          ↓
strip_dev_files() — second safety net
          ↓
Conflict check (update vs fresh install)
          ↓
Atomic move → /var/www/extensions/installed/<ext-id>/
          ↓
Set permissions: files 644, dirs 755, scripts 755
          ↓
Create PHP route: ln -s template.php /var/www/<ext-id>.php
          ↓
EXT_MGR_EXTENSION_ROOT=... sudo -u moode-extmgrusr bash scripts/install.sh
          ↓
Menu registration (manifest.menuVisibility → moOde)
          ↓
Extension active
```
