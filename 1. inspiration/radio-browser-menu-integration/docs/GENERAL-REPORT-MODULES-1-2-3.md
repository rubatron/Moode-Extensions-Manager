# General Report: Modules 1, 2, and 3

## Project Summary
This project integrated Radio Browser cleanly into moOde with a phased approach:
- Module 1: restore reliable Configure modal behavior on the Radio Browser page
- Module 2: add direct Library menu access to Radio Browser
- Module 3: add a flexible Extensions Manager with dynamic submenu behavior

Environment used:
- Host: `moode1.local`
- Platform: moOde 10.1.2 on Raspberry Pi Zero 2W
- Deployment tooling: SSH/SCP + shell installers

## Goals and Scope
Primary goals:
- Make Radio Browser behave consistently with core moOde UX
- Keep core modifications minimal and reversible
- Provide repeatable installer-based deployment
- Prepare the menu system for future extensions beyond Radio Browser

Out of scope for this phase:
- Full database migration for extension metadata
- Advanced extension lifecycle management (install/remove/version policy)

## Module 1: Modal Fix Integration
### Problem
On `radio-browser.php`, opening `#configure-modal` could show backdrop but not the modal itself.

### Implemented Solution
- Core condition patch in `header.php` so the menu behavior also applies to `radio-browser` context.
- Extension-side fallback script (`radio-browser-modal-fix.js`) to force modal open when the default flow stalls.
- Extension entrypoint update to load the fallback script.

### Result
- Configure modal behavior on `radio-browser.php` is restored and consistent with expected menu flow.

## Module 2: Library Menu Button
### Problem
Users needed quick access from the Library dropdown to Radio Browser.

### Implemented Solution
- Added `Radio Browser` button to Library menu in index template.
- Canonical route enforced to `/radio-browser.php`.

### Result
- One-click navigation to Radio Browser from Library menu.

## Module 3: Extensions Manager Foundation
### Objective
Move from one hardcoded button to a reusable extension menu architecture.

### Architecture Choice
- File-based registry (`/var/www/extensions/registry.json`) instead of immediate SQLite schema changes.
- Reason: lower coupling, faster rollout, easier rollback.

### Implemented Components
- `extensions-registry.php`: discovery and registry generation
- `extensions-manager.php`: manager UI page
- `extensions-manager-refresh.php`: refresh endpoint
- Root shortcuts:
  - `/extensions-manager.php`
  - `/extensions-manager-refresh.php`
- Dynamic menu integration in index template
- Hover/inline submenu script: `/extensions/extensions-hover-menu.js`

### UX Refinements Completed
- Extensions entry moved to first position in Library list.
- Inline submenu styling aligned with existing menu item sizing.
- Extension icon added per item.
- Pin button added per item with persisted state in `localStorage`.
- Pinned behavior updated so panel is visible when Library menu is opened (not only on hover).
- Pin icon reduced in visual weight and size.

### Route Correction (latest)
- Resolver priority updated so root entry points are preferred.
- `radio-browser` now resolves to `/radio-browser.php` (not `/extensions/installed/radio-browser/radio-browser.php`).

## Deployment and Validation
Installers:
- `scripts/install-modal-fix.sh`
- `scripts/install-module-2-menu-button.sh`
- `scripts/install-module-3-extension-manager.sh`

Validation performed during rollout:
- Remote syntax checks (`bash -n`, `php -l`)
- Render checks via `curl` and marker matching
- Registry output checks for correct extension entry URLs
- Live confirmation on `moode1.local`

## Risk and Rollback Strategy
Risk controls used:
- Minimal edits to core template flow
- Logic moved to extension-owned JS/PHP where possible
- Automatic template backups during installer execution

Rollback approach:
- Restore template backup generated by installer
- Reinstall previous module version if needed
- Rebuild registry via refresh endpoint

## Current Status
- Module 1: complete and stable
- Module 2: complete and stable
- Module 3: complete baseline with refined UX and corrected Radio Browser routing

Overall status: all three modules are operational, reproducible through installers, and aligned with the intended moOde menu behavior.
